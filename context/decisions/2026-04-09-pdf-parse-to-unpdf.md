# Decision: Replace pdf-parse with unpdf

## Context

The document-upload feature needed PDF text extraction. We initially picked `pdf-parse` (v2.4.5) because it's the most popular Node.js PDF library and has a clean API. It worked perfectly during local development and in vitest unit tests — extracted all 19 pages of the AIveda proposal in milliseconds.

## The problem

When we tried to deploy to Vercel, the `/api/upload` route returned **HTTP 500 immediately on every request**, even before the auth check could run. The error message from `vercel --prod --yes` was empty (`"Unexpected error. Please try again later."`). The response body was Vercel's static `500: This page couldn't load` page, served with `x-matched-path: /500` and a cacheable etag — meaning Vercel had rejected the function bundle entirely, not just hit a runtime exception.

We tried three fixes in sequence:

1. **`serverExternalPackages: ["pdf-parse", "pdfjs-dist", "mammoth"]` in next.config.ts.** Tells Next/Turbopack to skip bundling and let Node `require()` them at runtime. Build succeeded but the function still crashed because Vercel's NFT (Node File Trace) only noted the package directory existed without copying its contents.

2. **`outputFileTracingIncludes` glob to force NFT to copy `pdfjs-dist/**/*`.** Trace count jumped from 388 to 1330 files. New deploy error: `"The framework produced an invalid deployment package for a Serverless Function. Typically this means that the framework produces files in symlinked directories."` This is the pnpm symlink issue — pnpm puts real files in `.pnpm/` and symlinks `node_modules/<name>` to them. NFT can't follow these symlinks reliably for serverless function packaging.

3. **`.npmrc` `node-linker=hoisted` to force pnpm flat installs + reinstall everything.** No more symlinks. Build still succeeded locally. New deploy error: same "invalid deployment package" — now the function bundle was simply too big (pdfjs-dist + pdf-parse + mammoth + dependencies pushed past Vercel's hobby tier function size limits).

Each fix got us further but the underlying problem stayed: **`pdf-parse` is fundamentally a poor fit for Vercel serverless because it depends on the heavy `pdfjs-dist` library which expects a worker file, dynamic imports, and a forgiving filesystem layout — none of which Vercel's serverless runtime provides cleanly.**

## The decision

**Replace `pdf-parse` with `unpdf`.**

`unpdf` is a 2024 library from the unjs ecosystem (unjs/unpdf) **purpose-built for serverless and edge environments**. Key properties:

- **Zero dependencies.** No worker file. No native bindings.
- **Ships its own slim pdfjs build** as a single .mjs file optimized for edge runtimes (Cloudflare Workers, Vercel, Deno, Bun).
- **Drop-in API** for our use case: `getDocumentProxy(buffer) + extractText(pdf, { mergePages: true })` returns `{ totalPages, text }` — same shape as what we were already passing downstream.
- **Stays under any reasonable function size limit.**

## Implementation cost

Trivial.

- One file changed in `web/src/lib/extract.ts` (the PDF extraction function).
- Error mapping converted from typed exception checks (`PasswordException`, `InvalidPDFException`, `FormatError`) to message-pattern matching, because unpdf surfaces pdfjs errors as plain Errors. The error codes our frontend renders against (`pdf_encrypted`, `pdf_invalid`, `pdf_format_error`) stayed identical.
- Removed `pdf-parse` from dependencies, added `unpdf`.
- Removed `pdfjs-dist` and `pdf-parse` from `serverExternalPackages`. Removed `outputFileTracingIncludes` block entirely.
- All 86 tests still pass without modification (the test fixtures are still PDFs generated by pdfkit; the test assertions check filtered text content, not library-specific exception types).
- `next.config.ts` shrank from 35 lines to 11 lines.

Total diff: `4 files changed, 45 insertions(+), 98 deletions(-)`.

## Verification

After deploy, hit `/api/upload` directly with curl:
- Without auth: HTTP 401, body `{"error":"unauthorized","message":"Sign in to upload documents."}` ✅
- The function is now executing and reaching the auth check, instead of crashing on import.

## Lessons captured

1. **For PDF text extraction on Vercel/serverless: always use `unpdf`. Never `pdf-parse`.** Documented in `memory/reference_vercel_serverless_libs.md` as a reusable cheat sheet for any future file-handling work in this project.

2. **When a Vercel deploy fails silently with empty `"Unexpected error"`, query the Vercel API directly:** `curl -H "Authorization: Bearer $TOKEN" "https://api.vercel.com/v13/deployments/<id>?teamId=..."` and look at `readyStateReason`. The CLI swallows the real error.

3. **The "static 500 page" symptom is diagnostic:** if a Vercel function returns 500 with `x-matched-path: /500` and a cacheable etag, the function bundle was rejected at deploy time, not at request time. Treat it as a "your function never started" problem, not a runtime bug.

4. **Library choice is part of the architecture, not an implementation detail.** A library that works flawlessly on your laptop can be unshippable in production if the runtime constraints don't match its assumptions. Always check serverless compatibility before adopting any heavy native-Node library.
